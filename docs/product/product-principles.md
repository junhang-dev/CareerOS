# CareerOS 제품 원칙

CareerOS는 사용자를 대신해 무작정 지원하는 자동화 도구가 아니라, 사용자가 더 좋은 판단을 하도록 커리어 정보를 정리하고 실행 흐름을 이어주는 개인 운영체제다.

## 제품 철학

1. 사용자의 판단권을 보존한다.

   공고 추천, 적합도 분석, 문서 초안은 모두 보조 정보다. 최종 제출, 메시지 발송, 외부 계정 변경은 사용자의 명시적 승인 뒤에만 진행한다.

2. 커리어 자산을 먼저 쌓는다.

   좋은 지원 준비는 이력서 파일 하나가 아니라 경력, 프로젝트, 성과, 스킬 근거, 문서 버전의 조합에서 나온다. CareerOS는 지원 전에 자산을 구조화하는 경험을 핵심으로 둔다.

3. 공고는 일회성 링크가 아니라 추적 대상이다.

   같은 공고라도 원문, 요건, 상태가 바뀔 수 있다. 원문과 구조화 결과를 버전으로 보존해 분석과 지원 전략이 어떤 근거에서 나왔는지 추적한다.

4. 자동화보다 설명 가능성을 우선한다.

   분석 결과에는 점수보다 이유가 중요하다. 강점, 리스크, 빠진 근거, 다음 액션이 드러나야 한다.

5. 작은 단일 사용자 제품으로 시작한다.

   MVP는 싱글유저 흐름을 끝까지 연결하는 데 집중한다. 멀티유저 협업, 고도화 추천, 자동 제출은 핵심 루프가 검증된 뒤 확장한다.

## 핵심 사용자 루프

1. 사용자가 선호 조건과 탐색 프로필을 정한다.
2. 공고가 수집되거나 수동으로 등록된다.
3. 공고 원문이 구조화되고 이전 버전과 비교된다.
4. 사용자의 커리어 자산과 공고 요건을 대조한다.
5. 적합도, 갭, 보강 액션을 확인한다.
6. 사용자가 지원 준비 단위를 만들고 문서/전략을 승인한다.

## 현재 MVP에서 지켜야 할 기준

- 입력 검증과 에러 계약은 `memory`와 `postgres` 드라이버에서 동일해야 한다.
- 사용자 화면은 다음 행동을 알려줘야 한다.
- 수동 입력 경로는 외부 연동보다 먼저 안정적이어야 한다.
- 원본 데이터와 구조화 데이터는 함께 보존한다.
- 민감정보와 일반 도메인 데이터는 섞지 않는다.

## 당장 중요한 제품 질문

- 사용자가 가장 먼저 완성해야 할 자산은 프로필, 경력, 프로젝트, 문서 중 무엇인가?
- 적합도 점수보다 더 신뢰를 주는 설명 단위는 무엇인가?
- 공고 수집 자동화 전에 수동 등록 경험을 얼마나 편하게 만들어야 하는가?
- 지원 준비 화면은 문서 편집 도구가 되어야 하는가, 승인/체크리스트 도구가 되어야 하는가?
